## MACs

### Programa `mac_sha256.py`

Exemplo de utilização:

```bash
$ echo "seguranca de sistemas informaticos" >> ssi.txt
$ python mac_sha256.py setup chave.key
Chave gerada e guardada em: chave.key
$ python mac_sha256.py mac ssi.txt chave.key
MAC gravado em: ssi.txt.mac
$ python mac_sha256.py ver ssi.txt chave.key
True
$ echo "hello there" >> ssi.txt
$ python mac_sha256.py ver ssi.txt chave.key
False
```

### Programa `mac_sha256_attack.py`

Exemplo de utilização:

```bash
$ echo "http://www.super-secret.com/manage?id=1001&role=user&perm=read" >> url.txt
$ python mac_sha256.py setup chave.key
Chave gerada e guardada em: chave.key
$ python mac_sha256.py mac url.txt chave.key
MAC gravado em: url.txt.mac
$ python mac_sha256_attack.py url.txt "&admin=true"
Ataque concluído com sucesso!
Nova mensagem gravada em: url.txt.ext
Novo MAC forjado gravado em: url.txt.ext.mac
$ python mac_sha256.py ver url.txt.ext chave.key
True
```

#### Questão 01

O ataque de extensão de comprimento explorado pelo programa `mac_sha256_attack.py` baseia-se na vulnerabilidade da construção $H(key || message)$ no SHA-256. Para calcular o padding, considera-se primeiro o comprimento total da entrada no estado interno do hash. A chave gerada no ficheiro `mac_sha256.py` possui 32 bytes. O URL fornecido como mensagem original contém 58 bytes. A entrada total processada pelo algoritmo soma 90 bytes. O SHA-256 opera em blocos de 512 bits ou 64 bytes conforme a RFC 6234. Os primeiros 64 bytes preenchem o primeiro bloco. Os restantes 26 bytes ocupam o início do segundo bloco. O padding deve completar este segundo bloco até atingir 64 bytes (ou 128 bytes no total acumulado). Conforme a norma, o padding inicia-se com o bit `1` representado pelo byte `0x80`. Seguem-se os bytes de preenchimento `0x00`. Os últimos 8 bytes do bloco reservam-se para o comprimento da mensagem em bits. O número de bytes nulos intermédios é calculado pela subtração do tamanho ocupado (26 bytes), do separador (1 byte) e do campo de comprimento (8 bytes) ao tamanho do bloco (64 bytes). Isto resulta em 29 bytes de valor zero. O padding total acrescentado pelo `hashpumpy` compreende o byte `0x80`, os 29 bytes `0x00` e os 8 bytes do campo de comprimento. O tamanho total do padding injetado entre a mensagem original e a extensão `&admin=true` é de 38 bytes.

## Cifras Autenticadas

### Programa `pbenc_aes_ctr_hmac.py`

Exemplo de utilização:

```bash
$ echo "seguranca de sistemas informaticos" >> ssi.txt
$ python pbenc_aes_ctr_hmac.py enc ssi.txt
Introduza a pass-phrase para cifrar: ...
Ficheiro cifrado e autenticado: ssi.txt.enc
$ python pbenc_aes_ctr_hmac.py dec ssi.txt.enc
Introduza a pass-phrase para decifrar: ...
Ficheiro decifrado com sucesso: ssi.txt.enc.dec
$ cat ssi.txt.enc.dec
seguranca de sistemas informaticos
```

### Programa `pbenc_aes_gcm.py`

Exemplo de utilização:

```bash
$ echo "seguranca de sistemas informaticos" >> ssi.txt
$ python pbenc_aes_gcm.py enc ssi.txt
Introduza a pass-phrase para cifrar (GCM): ...
Ficheiro cifrado com AES-GCM: ssi.txt.enc
$ python pbenc_aes_gcm.py dec ssi.txt.enc
Introduza a pass-phrase para decifrar (GCM): ...
Ficheiro decifrado com sucesso: ssi.txt.enc.dec
$ cat ssi.txt.enc.dec
seguranca de sistemas informaticos
```

#### Questão 02

O programa `pbenc_aes_ctr_hmac.py` utiliza uma construção manual do tipo _Encrypt-then-MAC_. Este script concatena individualmente o _salt_ de 16 bytes e o _nonce_ de 16 bytes. Adiciona uma etiqueta HMAC-SHA256 com 32 bytes de comprimento. O criptograma final mantém o tamanho exato do texto limpo original. O incremento total de tamanho neste ficheiro é de 64 bytes fixos.

O programa `pbenc_aes_gcm.py` utiliza o modo _Galois/Counter Mode_, que é um esquema AEAD nativo. Esta implementação armazena um _salt_ de 16 bytes e um _nonce_ otimizado de 12 bytes. A etiqueta de autenticação (tag) gerada internamente pelo AES-GCM possui apenas 16 bytes. O criptograma e a tag são processados de forma atómica pela biblioteca. O incremento total de tamanho neste ficheiro é de 44 bytes fixos.

A variação de 20 bytes entre os dois ficheiros deve-se a escolhas de design criptográfico. O modo CTR puro exige um _nonce_ que preencha todo o bloco de 128 bits do AES. O modo GCM reserva 4 bytes do bloco para o seu contador interno de 32 bits, reduzindo o _nonce_ externo para 96 bits. O HMAC-SHA256 prioriza uma resistência elevada a colisões com uma tag de 256 bits. O GMAC do AES-GCM utiliza uma tag de 128 bits, oferecendo um equilíbrio superior entre segurança e eficiência de armazenamento.
